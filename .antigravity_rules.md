# ⚡ Antigravity Harness & Efficiency Protocol

이 파일은 사장님께서 AI(Antigravity) 최고 수준의 효율과 정밀도를 통제하기 위한 **'하네스 프로토콜(Harness Protocol)'** 지침서입니다. 앞으로 AI에게 태스크를 지시할 때 강력한 하네스 템플릿을 복사하여 지시하시면 오작동을 차단하고 최상의 결과를 반환받을 수 있습니다.

---

## 👨‍💻 [USER용] 마스터 프롬프트 (명령 템플릿)
이후 새로운 기능(테스트 추가, 타로 기능 등)을 요건 정의하실 때 복사/붙여넣기 하십시오. 시스템 구조와 제약 조건을 명확하게 분리해주어야 AI가 절대 기존 코드를 망가뜨리지 않습니다.

```markdown
### [Goal] (가장 중요한 최종 목표)
예: 새로운 "직장인 생존 유형 테스트"를 추가하고 싶어. 코드는 기존 도파민 테스트 컴포넌트들을 재활용해.

### [Context] (참고해야 하는 힌트 및 배경지식)
예: `testTypes.ts`에 신규 테스트를 등록해야 메인에 노출됨. 컴포넌트는 `client/src/pages/` 에 추가해줘.

### [Constraint] (토큰 최적화 및 훼손 방지를 위한 절대 제약조건)
1. 기존 `CoupangRecommend.tsx`의 로직과 헤더는 절대 건드리지 마. (완벽하게 동작 중임)
2. 파일 전체를 덮어쓰지 말고 반드시 변경점 부분만 `multi_replace` 로 수술해.
3. 작업 완료 후에는 구구절절 설명하지 말고 "완료되었습니다" 결과만 요약해.
```

## 🤖 [AI용] 내부 동작 제어 강제 지시문 (System Meta-Instruction)

이 파일이 존재하는 한, 나(Antigravity Agent)는 다음 작업 강령을 절대적으로 준수한다.

1. **Surgical Edit (외과수술식 편집)**
   - 멀쩡히 동작하는 기존의 코드를 무심코 지우거나 덮어쓰지 않는다.
   - 파일을 수정할 때 전체를 Rewrite(재작성)하지 않고 반드시 `multi_replace_file_content` 도구를 사용하여 정확히 타겟 라인만 스나이핑하여 교체한다. (토큰 최적화)

2. **Zero Regression (부작용 차단)**
   - 새로운 라이브러리를 설치하거나 서버 `listen` 포트와 같은 민감한 커맨드를 조작해야 할 때는, 명령어가 타겟 OS(Windows)에 맞는 명령어인지 한 번 더 스스로 검증(`Command Status` 체크 등)한다.
   - 기존의 컴포넌트를 재활용할 경우, 기존 디자인과 `tailwind` 체계를 그대로 상속받고 불필요한 인라인 스타일 제안을 삼간다.

3. **No Yap (수다 금지 & 행동 증명)**
   - 사장님의 요청에 답할 때 기술에 대한 TMI(Too Much Information), 뻔한 원리 설명을 자제한다.
- 오직 [원인] -> [사용한 도구] -> [해결 결과] 순으로 극도로 압축하여 보고한다.

## 🌐 4. Architecture Context Constraint (아키텍처 절대 암기 구역)

나(Antigravity Agent)는 이 프로젝트의 배포/인프라 구조를 다음과 같이 영구적으로 인지하고 행동한다. 절대 추측하거나 이 구조를 위반하는 명령어를 서버에 전송하지 않는다.

1. **프론트엔드 (Vercel)**
   - Next.js 웹사이트(mbtifinder.com)는 Vercel에 연동되어 있다.
   - GitHub `main` 브랜치에 코드가 푸시되면 Vercel이 알아서 `npm run build`를 수행하고 호스팅한다.
   - **[절대 금지]** 따라서 그 어떤 경우에도 오라클 클라우드 원격 서버 내부에서 `npm run build`나 `next build`를 직접 실행하지 않는다.

2. **백엔드/봇 워커 (Oracle Cloud - 144.24.87.131)**
   - 오라클 서버는 오직 `scripts/cms-bot/bot.js` (텔레그램 봇) 하나만을 24시간 돌리기 위한 경량(1GB RAM) 워커 인스턴스다.
   - PM2 프로세스 이름은 `mbti-cms-bot` 하나만 존재해야 한다.
   - 봇 코드가 변경된 경우에만 오라클 서버에서 `git pull` 후 `pm2 restart mbti-cms-bot`을 실행하며, 프론트엔드 코드가 변경된 경우에는 오라클 서버에 접속조차 할 필요가 없다.

3. **데이터 흐름 (CI/CD)**
   - 텔레그램 명령어 -> 오라클 `bot.js`가 응답 -> GitHub PR 생성 -> User가 Merge -> Vercel 자동 배포 (오라클 서버는 이 과정에서 단지 브릿지 역할만 함)
